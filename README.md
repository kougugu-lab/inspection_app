# AI 自動検査システム (AI Camera Auto Inspection System)

**バージョン**: 3.3.4  
**最終更新**: 2026-04-19

## 📌 これは何をするプログラム？
このプログラムは、**カメラとAI（人工知能）を使って、部品の欠品や個数を自動で検査するシステム**です。
例えば工場の生産ラインなどで、「部品が正しく付いているか？」「指定された個数あるか？」を、人間の目の代わりにAIがカメラの映像を見て素早く判定します。

---

## 💡 主な機能

1. **AIによる自動判定 (YOLO)**
   * 事前に学習させたAIモデルを使って、カメラに映った物体（部品など）を見つけ出します。
   * 「部品Aが2個、部品Bが1個あるか？」といった複数条件の組み合わせ（AND条件）で精度の高いOK/NG判定が可能です。

2. **ハードウェアとの連携 (GPIO)**
   * **トリガー入力**: 外部センサーやPLCからの信号（トリガー）をきっかけに撮影と検査をスタートします。
   * **パターン切り替え**: セレクターピンのON/OFF組み合わせで、製品種類に合わせた検査条件（パターン）を自動切替します。
   * **結果出力**: 検査結果に応じてGPIO出力（パトライト点灯・ブザー鳴動）を制御します。

3. **証拠画像の自動保存とCSVログ**
   * 検査した瞬間の画像（AI検出枠付き）と元画像（NG_RAW）を自動保存します。
   * 日々の検査結果はCSVファイルとして自動集計します（Excelで開けます）。

4. **パソコン(Windows)でのテスト（モックモード）**
   * Windowsなどで起動した場合は「仮想GPIOパネル」が立ち上がり、物理センサーなしで動作テストができます。

## 🔄 動作フロー（アプリの処理の流れ）

本システムは以下の流れで検査を自動実行します。ユーザーの設定操作とは独立した内部処理を示します。

```
起動
  │
  ├─ 設定ファイル読み込み (inspection_settings.json)
  ├─ 結果フォルダの自動生成 (results/)
  ├─ YOLOモデルのロード → バックグラウンドでウォームアップ推論
  ├─ カメラの初期化（複数カメラを並列オープン）
  └─ GPIO（トリガー・セレクター・出力）の初期化
          │
          ▼
  [トリガー待機ループ]
  トリガー信号の受信
          │
          ▼
  セレクターピンの状態を読み取り → 実行する「パターン」を決定
          │
          ▼
  バースト撮影（全カメラで複数回フレームを取得）
          │
          ▼
  AIモデルで推論（YOLOで物体検出クラス・個数を取得）
          │
          ▼
  条件判定（設定したパターンの条件をすべて満たすか？）
    ├─ OK → OK出力ピンをON、OK画像を保存
    └─ NG → NG出力ピンをON + ブザー、NG画像（枠付き + 元画像）を保存
          │
          ▼
  CSVログに追記
          │
          ▼
  全トリガーのサイクル完了 → コミット番号を+1
  次のトリガー待機へ戻る
```

## 👤 ユーザー操作ガイド

### ステップ1: 学習用画像の収集（撮影モード）
まずAIモデルを作成するための画像を集めます。
1. 画面上部の「**撮影モード**」ボタンをクリックします。
2. トリガーが入るたびに判定は行わず、設定されたバースト枚数分の画像を `results/images/REC` フォルダに保存します。
3. 収集した画像でAIモデルを学習させます（学習方法は別ドキュメントを参照）。

### ステップ2: 設定（詳細設定ボタン）
画面右側の「**詳細設定**」ボタンから各設定を行います。

| タブ | 設定内容 |
|---|---|
| システム設定 | AIモデルのパス、プレビューFPS、バースト回数など |
| カメラ設定 | 使用するカメラのインデックス、解像度、保存先 |
| GPIO設定 | トリガーピン・セレクターピン・出力ピンの割り当て |
| パターン設定 | 判定条件（何をいくつ検出したらOKか）の定義 |

> **Raspberry Pi で動作が重い場合**: 「システム設定」タブの「プレビュー更新レート」を `1` fps 以下に設定すると負荷を大きく下げられます。

### ステップ3: 検査の実行（検査モード）
1. 画面上部の「**検査モード**」ボタンをクリックします。
2. 「トリガー待機中」の状態でセンサー（GPIOトリガー）の信号を待ちます。
3. 信号が入ると自動で撮影・判定し、結果をGPIO出力します。
4. NG発生時は右パネルの履歴リストに記録されます。ダブルクリックで画像を確認できます。

### その他の操作
| ボタン | 機能 |
|---|---|
| ＋ / － | コミット番号を手動で増減 |
| 番号入力 | テンキーでコミット番号を直接入力 |
| 履歴リセット | 画面のNG履歴リストを消去 |
| 結果フォルダ | 画像保存先フォルダをファイルマネージャーで開く |
| ブザー停止 | NG出力・ブザーを強制停止して待機状態へ戻す |

## 📁 結果フォルダ構成

```
results/
├── images/
│   ├── OK/          # OK判定時の画像（AI検出枠付き）
│   ├── NG/          # NG判定時の画像（AI検出枠付き）
│   ├── NG_RAW/      # NG判定時の元画像（枠なし、原寸保存）
│   ├── SKIP/        # SKIP判定時の画像
│   └── REC/         # 撮影モードで保存した画像
├── csv/
│   └── inspection_results_YYYYMMDD.csv   # 日別の検査ログ
└── logs/
    └── app_YYYYMMDD.log                  # 日別の動作ログ
```

### 画像ファイルの命名規則

```
{判定}_{コミット番号}_{パターン名}_{カメラ名}_{トリガー名}_{信頼度}.jpg
例: OK_0001_パターン1_カメラ1_トリガー1_0.95.jpg
```

## 🗂️ ファイル構成とモジュールの役割

```
inspection_app/
├── main.py                    # 起動エントリーポイント
├── inspection_settings.json   # 設定ファイル（自動生成）
├── README.md
├── CHANGELOG.md
└── modules/
    ├── __init__.py
    ├── app.py          # メインロジック・メイン画面UI
    ├── constants.py    # 定数定義（バージョン・色・フォント・パス）
    ├── settings.py     # 設定ファイルの読み書き管理
    ├── hardware.py     # GPIO・カメラ・モックモードのハードウェア制御
    ├── dialogs.py      # 詳細設定ダイアログ（設定画面UI）
    └── widgets.py      # 共通UIコンポーネント（テンキー・ツールチップ等）
```

### 各モジュールの役割（プログラマ向け）

| ファイル | 役割 |
|---|---|
| `main.py` | アプリの起動エントリーポイント。`sys.path` を設定し `InspectionSystem` を起動する。フォルダ名に依存せず動作する。 |
| `app.py` | アプリの中核。メイン画面UIの構築、カメラプレビューループ、トリガー検知からAI推論・判定・保存までの一連の検査ロジックを担う。 |
| `constants.py` | バージョン情報、色、フォント、パスなどのアプリ全体で共通する定数を定義。 |
| `settings.py` | `inspection_settings.json` の読み込み・保存・デフォルト値マージを管理する `SettingsManager` クラス。 |
| `hardware.py` | GPIO入出力デバイス（`DigitalInputDevice` / `OutputDevice`）とモックモード用の `MockManager` を提供。GPIO非使用環境では自動的にモックに切り替わる。 |
| `dialogs.py` | 「詳細設定」ダイアログウィンドウ。カメラ設定・GPIO設定・パターン設定の各タブUIを実装。 |
| `widgets.py` | 複数箇所で再利用する部品（`TenKeyDialog` テンキー、`Tooltip` ツールチップ、`HelpWindow` ヘルプ画面、`create_card` カードフレーム）を提供。 |
